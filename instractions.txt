Your notes are moving in the right direction, but they are still mixing three different things:

1. Timetable data (teachers, subjects, classes).
2. School calendar configuration (periods, breaks, timings).
3. Scheduling rules and constraints.

If you don't separate these now, the engine will become impossible to maintain when a school asks for custom rules.

## 1. Core Entities

The system should not think in terms of "Teacher X teaches Geo 5 lessons".

The system should think in terms of **Teaching Assignments**.

### Teacher

| Field             | Example  |
| ----------------- | -------- |
| TeacherID         | T001     |
| Name              | John Doe |
| Availability      | Custom   |
| MaxLessonsPerDay  | 6        |
| MaxLessonsPerWeek | 30       |

### Subject

| Field             | Example   |
| ----------------- | --------- |
| SubjectID         | S001      |
| Name              | Geography |
| Examinable        | Yes       |
| CBCDefaultLessons | 5         |

### Class / Stream

| Field   | Example |
| ------- | ------- |
| ClassID | F1E     |
| Form    | 1       |
| Stream  | East    |

### Teaching Assignment

This is the most important table.

| Teacher | Subject   | Class | Lessons |
| ------- | --------- | ----- | ------- |
| John    | Geography | F1E   | 5       |
| John    | Business  | F1E   | 4       |
| Mary    | Biology   | F2N   | 5       |

The timetable engine schedules assignments, not teachers.

# 2. School Configuration

Every school can have different timings.

### Example

School starts: 8:00 AM

School ends: 4:00 PM

Periods per day: 8

Period length: 40 min

### Break Configuration

| Break       | After Period | Duration |
| ----------- | ------------ | -------- |
| Assembly    | Before P1    | 20 min   |
| Tea Break   | After P4     | 20 min   |
| Short Break | After P6     | 10 min   |
| Lunch       | After P7     | 40 min   |

These should be configurable.

Never hardcode breaks.

---

# 3. Teacher Preferences

Example:

Teacher John unavailable:

* Monday P1
* Monday P2
* Friday P8

Store as:

| Teacher | Day    | Period |
| ------- | ------ | ------ |
| John    | Monday | P1     |
| John    | Monday | P2     |
| John    | Friday | P8     |

When generating the timetable, those slots become forbidden.

# 4. CBC Rules

You mentioned:

> All examinable subjects have 5 lessons.

This should be a rule, not data.

Example:

```text
IF subject.examinable = true
THEN lessons_per_week = 5
```

Unless school overrides it.

Example:

| Subject     | Examinable | Lessons |
| ----------- | ---------- | ------- |
| Geography   | Yes        | 5       |
| Mathematics | Yes        | 5       |
| Agriculture | No         | 3       |

---

# 5. Hard Constraints

These can NEVER be violated.

### Teacher Clash

Teacher cannot teach two classes simultaneously.

### Class Clash

Class cannot have two lessons simultaneously.

### Availability

Teacher unavailable = no lesson.

### Weekly Lesson Count

Exactly the specified number of lessons.

### No Double Lesson (CBC)

If enabled:

```text
Geo Monday P1
Geo Monday P2
```

Invalid.

---

# 6. Soft Constraints

These improve quality but may be violated if necessary.

### Spread Lessons

Bad:

```text
Geo
Mon P1
Mon P2
Mon P3
Mon P4
Mon P5
```

Good:

```text
Mon P1
Tue P2
Wed P4
Thu P3
Fri P5
```

### Avoid Last Period

Some schools dislike Mathematics at last period.

### Avoid Teacher Overload

Example:

Teacher has 8 lessons in one day.

Bad.

Try spreading across week.

# 8. Scheduling Algorithm

Do not start with AI.

Do not start with genetic algorithms.

Most student projects fail because they jump to advanced algorithms too early.

Start with:

### Phase 1

Greedy Scheduler

Sort assignments by difficulty.

Example:

```text
Geo F1E 5 lessons
Bio F2N 5 lessons
Math F3S 5 lessons
```

Place hardest lessons first.

Check constraints.

Move to next.

### Phase 2

Backtracking

If conflict occurs:

```text
Undo
Try another slot
```

### Phase 3

Optimization

Only after Phase 1 and 2 work.

Possible techniques:

* Genetic Algorithm
* Simulated Annealing
* Tabu Search

# What is missing from your design?

Three critical things:

### Room Allocation

Example:

```text
Chemistry Lab
Computer Lab
Library
```

Two classes cannot use the same lab simultaneously.

### Subject Constraints

Example:

```text
Chemistry requires double lesson
```

or

```text
CBC subject cannot be double lesson
```

### Workload Rules

Example:

```text
Teacher max 6 lessons/day
Teacher max 30 lessons/week
```

Without these, the engine will generate mathematically correct timetables that schools reject.

My recommendation before writing code is to produce:

1. Use Case Diagram.
2. ER Diagram.
3. Class Diagram.
4. Constraint Document.
5. Database Schema.
6. Scheduling Algorithm Specification.

Only after those six documents are complete should implementation begin. That discipline will save months of rewrites later.
